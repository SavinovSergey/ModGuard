# Архитектура ModGuard

ModGuard — система модерации контента. Документ описывает целевую архитектуру: сервис модерации с детекцией 
токсичности и спама, асинхронной обработкой через брокер сообщений и разделением ответственности между 
классификацией и действиями по результатам. Документ можно править по мере уточнения требований.

---

## 1. Цели и границы

- **Назначение:** единый сервис модерации контента (комментарии, сообщения) с поддержкой нескольких клиентов
 и приближением к production-уровню.
- **Функции:** детекция токсичности и детекция спама (в перспективе — оффтоп, дубли, язык, персональные 
данные и т.д.).
- **Клиенты (демо и далее):** веб-сайт, Telegram-чаты; возможны другие (мобильное приложение, API партнёров).
- **Требования:** высокая нагрузка, модульность, возможность масштабирования и независимой смены политик по 
площадкам.

---

## 2. Высокоуровневая схема

```
[Клиенты: Web, Telegram, ...]
         |
         v
[Listeners / API]  -->  [RabbitMQ: очередь запросов]
                              |
                              v
                   [Backend классификации]
                   (токсичность и спам считаются параллельно)
                              |
                              v
                   [Postgres: результаты]   [Redis: кэш]
                              |
                              v
                   [RabbitMQ: очередь результатов]
                              |
                              v
                   [Action-сервисы по площадкам]
                   (Telegram Actions, Web Actions, ...)
```

- **Listeners / API** принимают события (новое сообщение, комментарий) и только ставят задачу в очередь запросов.
- **Backend классификации** — один тип сервиса: забирает задачи из очереди запросов, выполняет детекцию токсичности и 
спама **параллельно** (одновременно, асинхронно друг относительно друга), объединяет результаты, пишет в Postgres и в 
Redis (кэш), публикует одно сообщение в очередь результатов.
- **Очередь результатов** потребляют только **Action-сервисы**. Действие (удалить, предупредить, заблокировать, 
заблюрить) определяется в коде конкретного Action-сервиса для каждой площадки, а не в бэкенде классификации.

## 3. Компоненты

### 3.1. Клиенты

- **Веб-сайт** — демо-интерфейс: ввод текста, отправка на модерацию, отображение результата (в т.ч. через polling 
по `task_id`).
- **Telegram-чаты** — бот: сообщения из чата уходят на модерацию; по результату Action-сервис Telegram выполняет 
предупреждение, удаление и т.д. по правилам площадки.
- В будущем — другие клиенты (приложение, API), подключаемые через свои Listeners и Actions.

### 3.2. Listeners (приём событий)

- **Назначение:** получить событие «новое сообщение/комментарий», **проверить кэш по тексту** (см. п. 3.5.1); 
при промахе — сформировать сообщение для модерации и опубликовать его в очередь запросов. Не содержат логики 
классификации и не выполняют действия.
- **Примеры:**
  - **Telegram Listener** — получает апдейты от Telegram (long polling или webhook); для каждого сообщения 
  проверяет кэш; при промахе публикует в RabbitMQ запрос с `task_id`, ссылкой на чат/сообщение, текстом, метаданными.
  - **Web / API** — приём HTTP-запросов от фронта (или батчей), генерация `task_id`, проверка кэша; при промахе —
   публикация в очередь запросов; ответ клиенту — `task_id` для polling (при cache hit результат можно отдать 
   сразу или после записи лёгкой строки в Postgres).
- При масштабировании можно поднимать несколько инстансов Listeners; балансировка — по месту развёртывания 
(Nginx для API, несколько воркеров для Telegram и т.д.).

### 3.3. Брокер сообщений (RabbitMQ)

- **Одна очередь запросов:** задачи на модерацию (текст, источник, ссылка на сообщение/комментарий, `task_id`). 
Потребляет Backend классификации.
- **Одна очередь результатов:** результат модерации по `task_id` (оценки токсичности/спама, флаги, ссылка на 
сообщение, источник). Потребляют Action-сервисы.
- В будущем при разделении детекции токсичности и спама по разным сервисам можно ввести exchange и маршрутизацию 
по типам задач; контракт сообщений при этом сохраняется (агрегация по `task_id` при необходимости).

### 3.4. Backend классификации

- **Один сервис (один тип воркера/контейнера):** внутри модули «токсичность» и «спам» выполняются **параллельно** 
(асинхронно друг относительно друга, например через asyncio или пул потоков), затем результаты объединяются в один
ответ. Так сокращается время обработки одной задачи.
- **Предобработка текста:** своя для каждого типа задачи (токсичность vs спам). Например, для спама не удалять 
цифры и email; для токсичности — своя нормализация. Каждый модуль использует свой препроцессор.
- **Поток задач:** задачи приходят из очереди запросов; после параллельной обработки результат записывается в 
Postgres и (при необходимости) в Redis, затем одно сообщение публикуется в очередь результатов. Бэкенд не решает, 
какое действие выполнить — только возвращает «факты» (scores, флаги).
- **Масштабирование:** несколько контейнеров Backend; все подписаны на одну очередь запросов — распределение 
нагрузки обеспечивает брокер.

#### 3.4.0. Двухэтапная проверка токсичности и спама

И для токсичности, и для спама используется **двухэтапный пайплайн**: сначала быстрый пре-фильтр по жёстким 
правилам (regex), затем при необходимости — ML-модель. Это снижает нагрузку на тяжёлые модели и даёт мгновенный 
результат для очевидных случаев.

- **Токсичность:**
  - **Этап 1 — regex пре-фильтр:** проверка на явный мат и очевидную токсичность (реализация в 
  `app/models/toxicity/regex_model.py`). При срабатывании сразу возвращается результат «токсично» 
  (`toxicity_score = 1.0`, `is_toxic = true`), ML-модель не вызывается.
  - **Этап 2 — ML-модель:** для текстов, которые regex не пометил как токсичные, вызывается цепочка ML-моделей 
  (bert → rnn → tfidf → fasttext) с fallback при ошибках/таймаутах. Regex также входит в fallback-цепочку как 
  последний вариант при недоступности всех ML-моделей.
- **Спам:**
  - **Этап 1 — regex пре-фильтр:** проверка по жёстким паттернам спама (в т.ч. CAPS_WORD, см. п. 3.4.1). 
  При срабатывании возвращается результат «спам» без вызова TF-IDF модели.
  - **Этап 2 — TF-IDF модель:** тексты, не попавшие под regex, классифицируются моделью TF-IDF + ручные признаки.

Для батчей: сначала по всему батчу применяется regex; только для элементов без срабатывания regex вызывается ML 
(батчем), затем результаты объединяются в один массив в исходном порядке.

#### 3.4.1. Жёсткие правила спама (блокировка по паттернам)

- **Решение:** тексты, в которых найден хотя бы один из паттернов CAPS_WORD, классифицируются как спам и 
**подлежат блокировке**. Проверка выполняется в модуле спама до или вместе с моделью; при срабатывании правила 
результат «спам» возвращается без учёта оценки модели (например, `spam_score = 1.0`, `is_spam = true`). 
Действие по результату (удаление, мут и т.д.) определяется в Action-сервисах по площадке.
- **Паттерны** (реализация в `app/features/spam_features.py`):
  - **CAPS_WORD_DOUBLE_EXCL:** слово целиком капсом (минимум 5 букв) + сразу после него не менее 2 восклицательных
  знаков (например БЕСПЛАТНО!!, СКИДКА!!!).
  - **CAPS_WORD_6 (порог 7 букв):** слово из 7+ букв целиком капсом (например БЕСПЛАТНО, ПОДАРОК). Порог 7 
  уменьшает срабатывание на короткие аббревиатуры.
- **Итог:** система блокирует за спам любой текст, в котором найден один из этих паттернов.

#### 3.4.2. Жёсткие правила токсичности (пре-фильтр по мату)

- **Решение:** тексты, в которых найдено совпадение с регулярными выражениями **явного мата и очевидной токсичности**, 
классифицируются как токсичные без вызова ML-модели. Проверка выполняется в модуле токсичности на этапе 1 
(см. п. 3.4.0); при срабатывании возвращается результат «токсично» (например, `toxicity_score = 1.0`, `is_toxic = true`, 
при необходимости — типы из набора правил). Действие по результату (предупреждение, удаление, мут и т.д.) определяется
 в Action-сервисах.
- **Паттерны** (реализация в `app/models/toxicity/regex_model.py`): группы регулярных выражений по категориям — явный 
мат (обсценная лексика), оскорбительные и грубые слова. Поиск без учёта регистра.
- **Итог:** любой текст, в котором сработал хотя бы один паттерн, считается токсичным на этапе 1; для остальных 
текстов используется ML-модель (этап 2).

### 3.5. Хранение данных

- **Postgres:** постоянное сохранение результатов классификации (task_id, текст/хэш, источник, токсичность, спам, 
метаданные, таймстампы). Используется для аудита, аналитики, ответа на polling (GET по task_id) и для дообучения моделей.
- **Redis:** кэш результатов классификации. См. ниже состав ключа и значения, а также проверку кэша до постановки в 
очередь. Персистентность Redis (RDB/AOF) — для сохранения кэша между рестартами. При необходимости — сессии, rate limit.

#### 3.5.1. Проверка кэша до бэкенда

- **Где:** перед публикацией задачи в очередь запросов — в API или в Listener (сразу после приёма события/запроса).
- **Действие:** по тексту сообщения (или по нормализованному тексту) вычисляется ключ кэша; выполняется запрос в Redis. 
При попадании в кэш результат можно сразу вернуть клиенту (или отдать по task_id после лёгкой записи в Postgres, см. 
ниже) и **не ставить** задачу в очередь запросов. При промахе — задача публикуется в очередь, обрабатывается бэкендом, 
результат кэшируется и пишется в Postgres как обычно.
- Один и тот же текст даёт один и тот же результат из кэша независимо от источника и пользователя.

#### 3.5.2. Состав данных в Redis (кэш)

- **Ключ:** хэш текста (или нормализованного текста), с единой схемой нормализации для кэша. Опционально префикс, 
например `moderation:toxicity:spam:` или версия пайплайна для инвалидации при обновлении моделей.
- **Значение:** только оценки моделей — токсичность, спам и при расширении другие поля (оффтоп, язык и т.д.). Без 
привязки к источнику: не храним source, user_id, chat_id, message_id, task_id. В кэше не важно, откуда и от кого 
пришло сообщение — только текст и результат.

#### 3.5.3. Запись в Postgres при cache hit (вариант C — гибрид)

- **При промахе кэша (запрос ушёл в бэкенд):** в Postgres пишется полная запись: task_id, source, ref, текст или 
text_hash, оценки (toxicity, spam и др.), метаданные, таймстамп. Это основная таблица результатов и аудита «кто что 
запросил и что получил» по уникальным вычислениям.
- **При попадании в кэш:** в Postgres пишется только **лёгкая** запись для аудита: task_id, source, ref, text_hash, флаг 
`from_cache = true`, таймстамп; при необходимости — сами scores (если хотите хранить все выданные ответы в БД). Полный 
текст при cache hit можно не дублировать — он уже есть в записи первого запроса с этим text_hash (или только в Redis).
- Итог: полная история запросов и выданных ответов (в т.ч. «кто и когда получил ответ из кэша»), без избыточного 
дублирования полного текста и тяжёлых записей на каждый cache hit.

### 3.6. Action-сервисы (действия по результатам)

- **Назначение:** получать из очереди результатов сообщения с результатом модерации и выполнять действия, принятые для 
данной площадки.
- **Ответственность за политику:** что делать при «токсично» или «спам» — определяется только в коде конкретного 
Action-сервиса. Примеры:
  - **Telegram:** токсичное сообщение → предупреждение пользователю; повтор → мут/удаление; спам → удаление.
  - **Соцсеть для детей:** токсичное → сразу блокировка; спам → скрытие и репорт.
  - Другие площадки — свои правила в своих Actions.
- Каждый Action-сервис подписан на очередь результатов и фильтрует/обрабатывает сообщения по источнику (source) или 
по типу площадки. Идемпотентность обработки (по task_id/ref) — в зоне ответственности Action-сервиса.

### 3.7. API (оркестрация для веб-клиента)

- Приём запросов от веб-клиента (одиночных или батчей), генерация `task_id`, **проверка кэша по тексту** (п. 3.5.1); 
при промахе — публикация в очередь запросов, возврат `task_id`. При cache hit — возврат результата сразу.
- Эндпоинт получения результата: GET по `task_id` — чтение из Postgres (или из Redis как кэша) и отдача статуса и
результата (polling).
- При масштабировании — несколько реплик API за Nginx (или балансировщиком).

### 3.8. Nginx (опционально, перед API и фронтом)

- SSL-терминация.
- Rate limiting по IP (и при необходимости по path).
- Проксирование к API и к статике фронтенда.

### 3.9. Фронтенд (демо)

- Отдельный контейнер/сервис: веб-интерфейс для ввода текста, отправки на модерацию и отображения результата 
(в т.ч. асинхронно через polling по `task_id`). Батчирование на клиенте — см. п. 3.10.

### 3.10. Обработка батчами

Обработка батчами снижает число запросов к API и позволяет эффективно использовать модели на стороне бэкенда.

#### 3.10.1. Клиентское батчирование (например, веб-демо)

- **Буфер:** клиент накапливает запросы (пары «локальный id / текст») в памяти.
- **Отправка батча** срабатывает при **первом из двух** условий:
  1. В буфере накопилось **не менее 1000** запросов — отправляем батч сразу.
  2. Прошла **1 минута** с момента появления первого запроса в текущей «волне» — отправляем всё, что есть в буфере 
  (даже один запрос).
- После отправки буфер и таймер сбрасываются; следующий запрос начинает новую волну.
- Клиент получает от API один `task_id` на весь батч и по нему делает polling; при `completed` приходит массив 
результатов в том же порядке, что и элементы батча (сопоставление по индексу или по локальному id).

#### 3.10.2. Приём батча в API / Listener

- API может принимать как одиночный запрос, так и батч (список текстов с опциональными локальными id). Для батча 
генерируется **один** `task_id` на весь батч.
- **Кэш:** перед постановкой в очередь для каждого текста в батче проверяется кэш. Тексты, попавшие в кэш, не 
отправляются в бэкенд; их результаты подставляются в ответ. Только тексты с промахом кэша формируют одну задачу в 
очереди запросов (один батч — один message с полным списком items и общим task_id). При выдаче результата по 
task_id массив собирается: из кэша для hit, из результата бэкенда для остальных, порядок сохраняется.

#### 3.10.3. Сообщение в очереди и обработка в бэкенде

- **Одно сообщение в очереди запросов** может соответствовать **одному** тексту (одиночный запрос) или **батчу** текстов:
 список items, каждый с text, ref (и при необходимости source, локальный id), общий `task_id` для батча.
- **Backend** забирает сообщение; если это батч — обрабатывает все тексты **батчем**: проверка кэша по каждому (при 
проверке в бэкенде), вызов моделей токсичности и спама батчами, чтобы эффективно использовать GPU/CPU. Результаты — 
массив в том же порядке, что и входной батч; одна запись в Postgres на задачу (task_id + массив результатов); запись в 
Redis по каждому уникальному тексту; одно сообщение в очередь результатов с массивом и общим task_id.
- **Ограничение размера батча** на стороне API (например, макс. 1000 элементов на один запрос/одну задачу) и при 
необходимости разбиение клиентом или API на несколько задач при большем объёме.

#### 3.10.4. Результат по task_id и для Action-сервисов

- GET /tasks/{task_id} при `completed` возвращает массив результатов в том же порядке, что и запрос (или список с id, 
если клиент передавал локальные id).
- Сообщение в очереди результатов для батча содержит task_id, source и **массив** элементов (ref, scores для каждого); 
Action-сервисы итерируют по массиву и выполняют действие по каждому элементу согласно своей политике.

## 4. Контейнеры (Docker)

| Сервис | Назначение |
|--------|------------|
| **api** | FastAPI: приём запросов, постановка в очередь, GET по task_id |
| **backend** | Воркер классификации: потребляет очередь запросов, запускает токсичность + спам, пишет в Postgres/Redis, публикует в очередь результатов |
| **listener-telegram** | Приём событий Telegram, публикация в очередь запросов |
| **listener-web** | При необходимости отдельный сервис приёма веб-событий; иначе веб идёт через API |
| **actions-telegram** | Потребляет очередь результатов, выполняет действия в Telegram по своей политике |
| **actions-web** | Потребляет очередь результатов, выполняет действия для веб-платформы (обновление БД, флаги и т.д.) |
| **rabbitmq** | Брокер (очередь запросов и очередь результатов) |
| **postgres** | БД для результатов модерации и аудита |
| **redis** | Кэш (с персистентностью) |
| **nginx** | SSL, rate limit, прокси |
| **frontend** | Статика/приложение демо-интерфейса |

Масштабирование: несколько реплик `api`, несколько реплик `backend`, при необходимости несколько реплик `actions-*`.

## 5. Контракт сообщений (идея для реализации)

### 5.1. Сообщение в очередь запросов

- **Одиночный запрос:** одно сообщение — один текст.
- **Батч:** одно сообщение — массив элементов, общий `task_id` на весь батч (см. п. 3.10).
- Поля (для одного item или для каждого элемента батча):
  - `task_id` — уникальный идентификатор задачи (для батча один на весь батч).
  - `source` — источник (telegram, web, …).
  - `ref` — ссылка на объект (chat_id, message_id; или id комментария и т.п.) для последующего действия.
  - `text` — текст для модерации.
  - `options` — опционально (язык, тип контента и т.д.).
  - Метаданные (timestamp, user_id при наличии, локальный id клиента и т.д.).

### 5.2. Сообщение в очередь результатов

- **Одиночный результат:** одно сообщение — один набор scores.
- **Батч:** одно сообщение — массив результатов в том же порядке, что и запрос; общий `task_id`.
- Поля (на сообщение или на элемент батча):
  - `task_id` — тот же, что в запросе.
  - `source`, `ref` — для идентификации объекта действия (для батча — массив ref в том же порядке).
  - `toxicity_score`, `spam_score` (и при необходимости флаги `is_toxic`, `is_spam`).
  - Дополнительно: `toxicity_types`, метаданные модели, таймстамп.
- **Без поля «действие»:** решение о действии принимается только в Action-сервисе.

### 5.3. Ответ по GET /tasks/{task_id}

- Статус: `queued` | `processing` | `completed` | `failed`.
- При `completed` — результат в том же формате, что и в очереди результатов: для одиночного запроса — один объект
 с scores/флагами; для батча — массив результатов в том же порядке, что и запрос (для сопоставления по индексу 
 или по локальному id на клиенте).

## 6. Модульность и расширение

- **Новые типы модерации (оффтоп, дубли, язык):** добавляются как новые модули внутри Backend с собственными 
препроцессорами и моделями; результат расширяется полями в том же сообщении в очередь результатов. Action-сервисы 
при необходимости учитывают новые поля по своей логике.
- **Разделение Backend на отдельные сервисы (токсичность / спам):** при росте нагрузки можно вынести модули в 
отдельные воркеры и ввести exchange в RabbitMQ с маршрутизацией; агрегация по `task_id` и публикация одного 
сообщения в очередь результатов — в отдельном агрегаторе или в API.
- **Новые клиенты:** новый Listener + новый Action-сервис, подписка на ту же очередь результатов и своя политика 
в коде Actions.

## 7. Краткое резюме

| Элемент | Решение |
|--------|---------|
| Классификация | Один Backend: токсичность и спам считаются параллельно (асинхронно друг относительно друга), затем результат уходит в очередь результатов. |
| Двухэтапная проверка | Токсичность и спам: этап 1 — regex пре-фильтр (жёсткие правила), этап 2 — ML-модель только для текстов, не попавших под regex. Батч: сначала regex по всем элементам, затем ML по «промахам». |
| Токсичность: жёсткие правила | Тексты с совпадением по паттернам мата/очевидной токсичности (regex в `app/models/toxicity/regex_model.py`) сразу считаются токсичными без вызова ML. |
| Спам: жёсткие правила | Тексты с паттернами CAPS_WORD (слово капсом + !! или слово 7+ букв капсом) считаются спамом и блокируются без вызова TF-IDF. |
| Действия | В коде конкретного Action-сервиса для каждой площадки (Telegram, веб для детей и т.д.). |
| Очереди | Две: запросы (в Backend) и результаты (в Action-сервисы). |
| Хранение | Postgres — результаты и аудит; Redis — кэш (только текст как ключ + оценки). При cache hit — проверка до очереди; в Postgres лёгкая запись (вариант C). |
| Предобработка | Своя для токсичности и для спама (например, для спама не удалять цифры и email). |
| Масштабирование | Несколько контейнеров API, Backend и Action-сервисов; брокер распределяет нагрузку по очереди запросов. |
| Батчи | Клиент: отправка при 1000 запросов или раз в 1 мин. API/очередь: один task_id на батч. Backend: обработка батчами для эффективности моделей. Результат — массив в том же порядке. |

Документ можно дополнять и править в процессе рефакторинга.
