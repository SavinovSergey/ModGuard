# Дизайн-документ ModGuard

ModGuard — система модерации контента для автоматической классификации токсичности и спама в комментариях и сообщениях. 
Документ описывает истоки, проблематику, цели, требования и план развития системы.

---

## 1. Истоки

SocialConnect - это крупнейшая социальная сеть в России и странах СНГ, объединяющая миллионы пользователей.

Бизнес-модель: SocialConnect действует как платформа для социального взаимодействия, предоставляя пользователям 
возможность общения, обмена контентом, создания сообществ и групп.
Компания монетизирует платформу через рекламу, премиум-подписки, виртуальные подарки и партнерские программы.

География: Работает в России, Беларуси, Казахстане, Кыргызстане, Армении, Узбекистане, Азербайджане и других странах СНГ.
В 2024 году объявила о планах расширения на рынки Восточной Европы.

Функционал: Новостная лента, личные сообщения, группы и сообщества, видеохостинг, стриминг, игры, маркетплейс, 
платежи и переводы.

Ключевые показатели (по данным на 2023/2024 гг.):
* Ежемесячная активная аудитория: 100+ млн пользователей
* Ежедневная активная аудитория: 60+ млн пользователей
* Доля на рынке социальных сетей РФ: около 45%
* Выручка: 120+ млрд рублей

Пользовательская активность: сотни миллионов постов и комментариев в день, обработка до 50 млн комментариев ежедневно.

2024-2025: SocialConnect внедряет ИИ-решения для модерации контента и улучшения пользовательского опыта.
Система модерации ModGuard охватывает детекцию токсичности и детекцию спама в едином пайплайне.
Существующие решения: В экосистеме есть проекты анализа тональности и базовой модерации,
но специализированной системы модерации (токсичность + спам) с высокой точностью и единым API не было.

## 2. Проблематика

Текущее состояние:
* ~50,000,000 новых комментариев в день.
* Существующие системы модерации фокусируются на явном мате и примитивном спаме, но не на тонких формах токсичности
и разнообразных видах спама.

Проблемы (токсичность):
* Токсичные комментарии ухудшают пользовательский опыт и создают негативную атмосферу в сообществе.
* Троллинг, кибербуллинг и оскорбления отталкивают пользователей от платформы.
* Ручная модерация не масштабируется на такие объемы контента.
* Токсичность может проявляться в различных формах: явные оскорбления, скрытые намеки, сарказм, агрессивная критика.
* Пользователи используют обфускацию (замена символов, эмодзи, сленг) для обхода базовых фильтров.
* Контекстная токсичность: комментарий может быть токсичным только в определенном контексте обсуждения.

Проблемы (спам):
* Спам-сообщения, реклама и накрутка засоряют ленту и чаты, ухудшая пользовательский опыт.
* Массовые рассылки, ссылки на подозрительные ресурсы и промо-контент требуют автоматической фильтрации.
* Спам эволюционирует (новые форматы, обход фильтров), необходимы модели и правила, адаптируемые под политику площадки.

## 3. Цели и задачи проекта

Основная цель: автоматическая модерация контента (классификация токсичности и спама) с высокой точностью для 
улучшения качества платформы.

Задачи:
* Классификация комментариев/сообщений на «токсичные» / «нетоксичные» с вероятностной оценкой.
* Классификация на «спам» / «не спам» с вероятностной оценкой; единый ответ модерации содержит оба типа оценок.
* Определение типов токсичности (оскорбления, угрозы, троллинг, агрессия, ненависть и т.д.)
* Обработка обфускаций и эвфемизмов (скрытые формы токсичности и спама).
* Контекстный анализ (различение токсичности в цитатах vs прямом использовании, сарказм).
* Учет контекста обсуждения (токсичность может зависеть от темы и предыдущих сообщений).
* Интеграция через единый API и очереди для использования системой модерации и другими сервисами (веб, Telegram и т.д.).

## 4. Функциональные требования

Основной функционал:
* Классификация комментария/сообщения по токсичности (токсичный/нетоксичный) с бинарным результатом и вероятностью (0-1).
* Классификация по спаму (спам/не спам) с бинарным результатом и вероятностью (0-1); единый ответ модерации содержит
поля по токсичности и по спаму.
* Классификация типов токсичности (многоклассовая или мультилейбл): оскорбления, угрозы, троллинг, агрессия, ненависть,
дискриминация.
* Обработка различных форматов (текст, эмодзи, спецсимволы, Unicode, сленг).
* Поддержка русского языка с учетом региональных особенностей и интернет-сленга.
* Batch-обработка для больших объемов (массивы комментариев); один запрос может возвращать и оценки токсичности, и 
оценки спама.
* REST API и асинхронная обработка через очереди для интеграции с системой модерации и другими сервисами.
* Опциональная обработка контекста (предыдущие сообщения в ветке обсуждения).

API требования:
* Endpoint для одиночной классификации (модерация: токсичность + спам): POST /api/v1/classify
* Endpoint для batch-обработки: POST /api/v1/classify/batch-async
* Endpoint для получения статистики: GET /api/v1/stats
* Поддержка аутентификации (API keys)
* Rate limiting для защиты от злоупотреблений
* Документация API (OpenAPI/Swagger)

Дополнительный функционал:
* Логирование всех предсказаний для анализа и улучшения моделей (токсичность и спам).
* Возможность получения обратной связи (feedback loop) для переобучения.
* Экспорт результатов в различных форматах (JSON, CSV).

## 5. Нефункциональные требования

Производительность:
* Обработка 50,000,000+ комментариев в день (пиковая нагрузка ~1000 комментариев/сек)
* Латентность: < 100ms на один комментарий (для real-time модерации, p95)
* Batch-обработка: минимум 1000 комментариев за запрос
* Throughput: минимум 500 комментариев/сек на одну инстанцию

Точность и качество:
* По токсичности: Precision > 90%, Recall > 85%, F1 > 0.87; Average Precision (AP) как основная метрика при обучении.
* По спаму: Precision > 95%, Recall > 95%, F1 > 0.95.
* Оптимальный порог классификации подбирается автоматически после обучения (для токсичности и спама — максимизация F1 при Precision >= 90%).
* Метрики должны быть достигнуты на независимой тестовой выборке.
* Отдельные метрики для каждого типа токсичности (если используется многоклассовая классификация).

Надежность и доступность:
* Доступность: 99.9% uptime (менее 8.76 часов простоя в год)
* Отказоустойчивость: graceful degradation при сбоях модели
* Резервное копирование данных и конфигураций
* Мониторинг здоровья системы (health checks)

Масштабируемость:
* Горизонтальное масштабирование (возможность добавления инстанций)
* Вертикальное масштабирование (увеличение ресурсов инстанции)
* Автомасштабирование на основе нагрузки
* Поддержка распределенной обработки

Безопасность:
* Защита персональных данных (соответствие 152-ФЗ)
* Шифрование данных в покое и при передаче (HTTPS/TLS)
* Аутентификация и авторизация доступа к API
* Логирование доступа для аудита
* Защита от DDoS-атак

Совместимость:
* Поддержка Python 3.8+
* REST API совместим с любыми HTTP-клиентами
* Кроссплатформенность (Linux, Windows, macOS для разработки)

## 6. Ограничения

Технические ограничения:
* Зависимость от качества размеченных данных (garbage in, garbage out) — для токсичности и для спама.
* Вычислительные ресурсы для обучения и инференса (GPU для трансформеров).
* Сложность обработки очень длинных комментариев (> 10,000 символов).

Ограничения модели (токсичность):
* Ложные срабатывания на сленг/жаргон, сарказм, иронию, не являющиеся токсичностью.
* Сложность различения токсичности в контексте (цитаты, обсуждение новостей, критика).
* Необходимость регулярного переобучения из-за эволюции языка и появления новых форм токсичности.
* Субъективность понятия токсичности (разные люди могут по-разному оценивать один и тот же комментарий).

Ограничения модели (спам):
* Эволюция форматов спама и обход правил; необходимость обновления правил и переобучения модели.
* Различие определений спама по площадкам (реклама vs промо); гибкая настройка порогов и политик.

Бизнес-ограничения:
* Необходимость постоянного мониторинга и обновления моделей (токсичность и спам).
* Затраты на инфраструктуру и поддержку.
* Зависимость от внешних сервисов (API SocialConnect, облачные провайдеры).

Юридические ограничения:
* Соответствие требованиям законодательства о персональных данных.
* Этические вопросы автоматической модерации контента.
* Прозрачность работы алгоритма для пользователей.

## 7. Техническая архитектура (кратко)

Детальная архитектура системы ModGuard (очереди, Backend классификации, Listeners, Action-сервисы, кэш, батчи) описана в [ARCHITECTURE.md](ARCHITECTURE.md).

Общая архитектура:
* Микросервисная архитектура для масштабируемости.
* Разделение на компоненты: API / Listeners, Backend классификации (токсичность + спам), очереди, хранилище, 
Action-сервисы по площадкам.

Модели классификации:
* Токсичность: предварительный фильтр регулярными выражениями; классический ML (TF-IDF, FastText); RNN; 
fine-tuning трансформера (rubert-tiny и др.); предобработка и подбор порога по максимизации F1 при precision ≥ 0.9.
* Спам: правила (например, CAPS_WORD), TF-IDF + ручные признаки (капс, URL, email и т.д.); отдельная предобработка для
спама (сохранение регистра и ссылок где нужно).
* Оба типа считаются в Backend параллельно; результат объединяется в один ответ модерации.

Инфраструктура:
* API: FastAPI; очереди: RabbitMQ; БД: PostgreSQL (результаты, аудит), Redis (кэш).
* Деплой: Docker, при необходимости Kubernetes или Docker Compose.

## 8. Метрики успеха

Технические метрики:
* Токсичность: Precision > 90%, Recall > 85%, F1 > 0.87; AP > 0.90 как основная метрика при обучении; оптимальный порог 
по максимизации F1 при Precision >= 90%.
* Спам: Precision > 95%, Recall > 95%, F1 > 0.95; отдельный подбор порога для спама (максимизация F1 при precision >= 0.9).
* Confusion matrix и метрики по типам токсичности при многоклассовой классификации.

Бизнес-метрики:
* Снижение количества жалоб на токсичный и спам-контент.
* Снижение времени модерации комментариев.
* Улучшение пользовательского опыта; успешная интеграция с веб и Telegram (и другими площадками).

Производительность:
* Throughput и латентность в соответствии с п. 5; батчи до 1000 комментариев.

Качество данных:
* Достаточный объем и разнообразие размеченных данных для токсичности и для спама; стратифицированное разбиение 
train/val/test.

## 9. Риски

Риски данных:
* Низкое качество или недостаточный объем размеченных данных (токсичность/спам) → валидация, очистка, data augmentation,
предобученные модели.
* Смещение в данных (bias) → анализ распределения, стратифицированная выборка.

Риски модели:
* Переобучение, недообучение, дрифт → кросс-валидация, регуляризация, continuous learning, мониторинг метрик.

Технические риски:
* Высокая латентность → оптимизация (квантизация, кэш, батчи), масштабирование.
* Сбои инфраструктуры → резервное копирование, health checks, автоматическое восстановление.

Бизнес-риски:
* Изменение языка/сленга/форм токсичности и спама → обновление датасетов, переобучение, гибкие пороги и политики в 
Action-сервисах.

Юридические риски:
* Персональные данные, этика автоматической модерации → соответствие 152-ФЗ, прозрачность, возможность обжалования.

## 10. План развития

Фаза 1: MVP системы модерации ModGuard (токсичность + базовый спам)
* Базовая модель классификации токсичности и модель/правила для спама на размеченных данных.
* Единый REST API с ответами по токсичности и спаму; базовая предобработка для обоих типов.
* Документация API; целевые метрики по токсичности (Precision > 85%, Recall > 80%); приемлемое качество по спаму.

Фаза 2: Оптимизация производительности и точности
* Улучшение моделей (fine-tuning трансформеров для токсичности, доработка спама — фичи, пороги).
* Batch-обработка, кэширование, оптимизация инференса; мониторинг и логирование.
* Достижение целевых Precision > 90%, Recall > 85% по токсичности; улучшение метрик по спаму.

Фаза 3: Расширенная функциональность
* Многоклассовая классификация типов токсичности; при необходимости — расширение типов модерации (оффтоп, дубли, язык).
* Обработка контекста обсуждения (опционально); feedback loop для continuous learning.
* Расширенная аналитика и отчётность; поддержание метрик и адаптация политик по площадкам.

## 11. Датасет и разметка

Текущее состояние:
* Размеченные комментарии для классификации токсичности имеются в наличии.
* Для спама используются отдельные датасеты и разметка; при необходимости описать структуру в документации по обучению.

Требования к датасету (токсичность):
* Минимальный размер: 100,000 размеченных комментариев для обучения.
* Баланс классов: соотношение токсичных/нетоксичных близко к реальному (примерно 10-20% токсичных).
* Разнообразие: типы контента, темы, стили, типы токсичности.

Структура данных:
* Текст комментария; метка класса (токсичный/нетоксичный); опционально — типы токсичности, контекст обсуждения.
* Для спама: текст, метка спам/не спам; при необходимости — признаки (ссылки, капс и т.д.).

Разделение: train/val/test (например, 70/15/15), стратифицированное по классам.

Датасеты для [токсичности](https://huggingface.co/datasets/Mnwa/russian-toxic) и [спама](https://huggingface.co/datasets/benzlokzik/russian-spam-fork).

## 12. Конкуренты и аналоги

Существующие решения (токсичность):
* Встроенные системы модерации социальных сетей (VKontakte, Telegram, Facebook) — закрытые решения.
* Общие библиотеки фильтрации токсичности (detoxify, Perspective API) — не всегда учитывают специфику русского языка 
и единый пайплайн токсичность+спам.
* Коммерческие и open-source решения — требуют доработки под единую систему модерации.

Существующие решения (спам):
* Встроенные антиспам соцсетей и мессенджеров (VKontakte, Telegram, WhatsApp) — закрытые, без доступа к порогам 
и правилам.
* Спам-фильтры почтовых провайдеров (Gmail, Яндекс.Почта) — ориентированы на email, не на комментарии и посты.
* Akismet, reCAPTCHA и др. — защита от спама в блогах и формах; часто требуют интеграции с внешним API и не дают
единого пайплайна с токсичностью.
* Apache SpamAssassin и подобные — правила + ML в основном для почты; не заточены под русскоязычный контент в соцсетях.
* Узкоспециализированные антиспам-модули для комментариев — редко объединены с детекцией токсичности и с единым API 
для модерации.

Преимущества ModGuard:
* Специализация на русском языке и интернет-сленге.
* Единая система модерации: токсичность и спам в одном API и пайплайне.
* Высокая точность за счёт fine-tuning и отдельной предобработки под каждую задачу.
* Гибкая настройка порогов и политик по площадкам (Action-сервисы).
* Открытость и возможность кастомизации.
